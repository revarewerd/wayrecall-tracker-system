## 2. Проблемы и вопросы для обсуждения

### 🔴 Проблема 1: SSH и динамический IP
**Ситуация:** Если ограничить SSH по IP `95.165.68.186`, то подключиться можно только из дома. При подключении с работы, кафе, телефона — доступа не будет.

**Варианты решения:**
| Вариант | Плюсы | Минусы |
|---------|-------|--------|
| **A. Не ограничивать SSH по IP** | Подключение откуда угодно | Открыт для brute-force (но есть ключ + fail2ban) |
| **B. fail2ban + SSH ключи (без пароля)** | Безопасно + гибко | Нужен ключ на каждом устройстве |
| **C. VPN (WireGuard)** | Максимальная безопасность | Сложнее настройка, нужен VPN клиент везде |
| **D. Port knocking** | SSH порт скрыт до "стука" | Неудобно, ненадёжно |
| **E. SSH по ключу + ограничение по GeoIP** | Блокирует 95% ботов | Не точно, может заблокировать легитимный доступ |

**Мой рекомендуемый план:** Вариант B — отключить парольную аутентификацию SSH, оставить только ключи + fail2ban. Это даёт безопасность без привязки к IP.

### 🟡 Проблема 2: Мониторинг (Grafana, Prometheus)
**Ситуация:** Grafana (3000) и Prometheus (9090) — нужны только мне для отладки.

**Варианты:**
| Вариант | Описание |
|---------|----------|
| **A. Не пробрасывать на роутере** | Доступны только из LAN → я вижу их только дома |
| **B. SSH туннель** | `ssh -L 3000:localhost:3000 server` — безопасно, доступно откуда угодно |
| **C. Nginx reverse proxy + Basic Auth** | Доступ через 80/443 + пароль → но это +1 точка входа |

**Рекомендация:** Вариант A+B — не пробрасывать порты на роутере, а при удалённом доступе использовать SSH-туннель.

### 🟡 Проблема 3: Базы данных
**Ситуация:** TimescaleDB (5433) и Redis (6380) — нужны для отладки с ноутбука.

**Варианты:**
| Вариант | Описание |
|---------|----------|
| **A. Только Docker internal** | Сервисы обращаются через Docker network, порты не открыты на хосте |
| **B. Открыты на хосте, но LAN only** | Можно подключиться с ноутбука дома |
| **C. SSH туннель** | `ssh -L 5433:localhost:5433 server` → безопасно |

**Рекомендация:** B для дома + C для удалённого доступа. На роутере НЕ пробрасывать 5433/6380.

### 🟢 Проблема 4: GPS порты (5001-5004)
**Ситуация:** Трекеры подключаются с произвольных IP. Ограничить нельзя.

**Меры защиты:**
- Rate limiting на TCP уровне (Token Bucket в CM)
- Whitelist IMEI — неизвестные устройства отклонять
- Connection timeout — 60 сек без данных → disconnect
- Max connections per IP — защита от flood

### 🟢 Проблема 5: Web UI (80/443)
**Ситуация:** Должен быть доступен заказчикам для демо.

**Меры защиты:**
- Nginx reverse proxy с rate limiting
- HTTPS (Let's Encrypt) — потом, когда будет домен
- CORS правила
- API Gateway с аутентификацией

## 3. Предлагаемый план настройки (по шагам)

### Шаг 1: SSH hardening (ПЕРЕД фаерволом!)
```bash
# 1. Убедиться что ключ работает
ssh -p 2220 wogulis@192.168.1.5  # должно пустить без пароля

# 2. Отключить парольную аутентификацию
sudo nano /etc/ssh/sshd_config
# PasswordAuthentication no
# ChallengeResponseAuthentication no
# UsePAM no

# 3. Перезапуск sshd
sudo systemctl restart sshd

# 4. ПРОВЕРКА: попробовать подключиться заново (в отдельном терминале!)
```

### Шаг 2: fail2ban
```bash
sudo apt install fail2ban -y
# Настроить jail для SSH: 
# maxretry=3, bantime=3600, findtime=600
```

### Шаг 3: UFW (после подтверждения SSH)
```bash
# Сбрасываем
sudo ufw --force reset
sudo ufw default deny incoming
sudo ufw default allow outgoing

# SSH — открыт для всех (защита через ключи + fail2ban)
sudo ufw allow 2220/tcp comment 'SSH'

# GPS — открыт для всех
sudo ufw allow 5001:5004/tcp comment 'GPS trackers'

# Web — открыт для всех
sudo ufw allow 80/tcp comment 'HTTP'
sudo ufw allow 443/tcp comment 'HTTPS'

# Мониторинг — только LAN (удалённо через SSH-туннель)
sudo ufw allow from 192.168.1.0/24 to any port 3000 proto tcp comment 'Grafana LAN'
sudo ufw allow from 192.168.1.0/24 to any port 9090 proto tcp comment 'Prometheus LAN'

# БД — только LAN
sudo ufw allow from 192.168.1.0/24 to any port 5433 proto tcp comment 'TimescaleDB LAN'
sudo ufw allow from 192.168.1.0/24 to any port 6380 proto tcp comment 'Redis LAN'
sudo ufw allow from 192.168.1.0/24 to any port 29092 proto tcp comment 'Kafka LAN'

# Включаем
sudo ufw --force enable
```

### Шаг 4: Удалённый доступ к мониторингу (SSH-туннель)
```bash
# С ноутбука (откуда угодно):
ssh -p 2220 -L 3000:localhost:3000 -L 9090:localhost:9090 -L 5433:localhost:5433 -L 6380:localhost:6380 wogulis@95.165.68.186

# Теперь открываем:
# http://localhost:3000  → Grafana
# http://localhost:9090  → Prometheus
# localhost:5433         → TimescaleDB (DBeaver/psql)
# localhost:6380         → Redis (redis-cli)
```
