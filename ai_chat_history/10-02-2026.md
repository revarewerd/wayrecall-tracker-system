## Как работает вход в админку (биллинг) — простым языком

Представь себе **ночной клуб с охранником**. Вот аналогия:

### Шаг 1: Ты открываешь дверь (браузер → login.html)

Ты вводишь в браузере `http://127.0.0.2:5193/billing/`. Spring Security видит: "Этот человек не авторизован!" — и перекидывает тебя на страницу входа login.html.

Это настроено в security-app-context.xml:
```xml
<bean id="authenticationEntryPoint" ...>
    <constructor-arg value="/login.html"/>
</bean>
```

### Шаг 2: Ты вводишь логин и пароль (форма → JS)

На странице login.js рисуется ExtJS-форма с тремя полями:
- **Имя** (username)
- **Пароль** (password)  
- **Запомнить меня** (чекбокс)

### Шаг 3: Нажимаешь "Войти" (JS → HTTP запрос на сервер)

В login.js JavaScript отправляет POST-запрос:

```
POST /billing/j_spring_security_check
Данные: username=12345, password=12345
```

Это как ты подходишь к охраннику и показываешь ему паспорт.

### Шаг 4: Охранник проверяет тебя (Spring Security — цепочка проверок)

Вот тут самое интересное. В security-app-context.xml настроена **цепочка из 3 "охранников"** (authentication providers), которые проверяют тебя **по очереди**:

```xml
<sec:authentication-manager>
    <!-- Охранник 1: проверяет по файлу billingAdmins.properties -->
    <sec:authentication-provider ref="daoAuthenticationProvider"/>
    <!-- Охранник 2: проверяет по базе MongoDB -->
    <sec:authentication-provider ref="billingAuthenticationProvider"/>
    <!-- Охранник 3: бэкдор (вход под чужим аккаунтом) -->
    <sec:authentication-provider ref="backdoorAuthenticationProvider"/>
</sec:authentication-manager>
```

**Охранник 1 — файл `billingAdmins.properties`** ([строка 82-90](legacy-stels/monitoring/src/main/webapp-billing/WEB-INF/security-app-context.xml#L82)):

Это "список VIP-гостей", захардкоженный в файле:
```
12345=12345,DealerBackdoor,ROLE_USER,ROLE_ADMIN,...
```
Формат: `логин=пароль,роль1,роль2,...`

Spring читает этот файл прямо с диска (`${WAYRECALL_HOME}/conf/wrcinstances/default/billingAdmins.properties`), загружает в память, и просто **сравнивает строки**: введённый логин == строка до `=`, введённый пароль == первое значение после `=`.

Если совпало — **ты внутри**, дальше не проверяет.

**Охранник 2 — MongoDB** (BillingAuthenticationProvider.scala):

Если в файле тебя нет, запрос идёт в базу данных. В BillingAuthenticationProvider.scala происходит:

1. Ищет в коллекции `users` запись с `name == "введённый_логин"`
2. Достаёт из неё пароль
3. Spring сравнивает пароль (введённый vs из базы)
4. Затем подгружает **права** пользователя из коллекций `billingPermissions` и `billingRoles`
5. Перед впуском ещё проверяет — не заблокирован ли пользователь (BillingAuthenticationProvider.scala)

### Шаг 5: Тебя пускают или нет

**Если пароль верный:**
- Сервер создаёт **сессию** (JSESSIONID — как браслет на руке в клубе)
- Отправляет ответ `{ success: true }`
- JS делает `window.location = "/billing"` — перекидывает на главную

**Если пароль неверный:**
- Ответ `{ success: false, message: "Ошибка авторизации" }`
- Показывается окно с ошибкой

### Шаг 6: Дальнейшие запросы

После входа браузер при **каждом** запросе отправляет cookie `JSESSIONID`. Сервер проверяет по ней: "а, это тот чувак с браслетом, пускаем". Пароль заново не спрашивают.

## А что такое "фронт" вообще?

**Фронтенд — это НЕ сервис.** Фронтенд — это **код, который работает В БРАУЗЕРЕ пользователя**.

```
┌─────────────────────────────────────────────────────┐
│  Твой компьютер (или телефон)                       │
│                                                     │
│  ┌─────────────────────────────────────────┐        │
│  │ Браузер (Chrome)                        │        │
│  │                                         │        │
│  │  HTML → рисует кнопки, таблицы          │        │
│  │  CSS  → красит их, делает красиво       │  ← ЭТО │
│  │  JS   → оживляет (клики, запросы)       │  ФРОНТ │
│  │                                         │        │
│  └──────────────┬──────────────────────────┘        │
└─────────────────┼───────────────────────────────────┘
                  │ HTTP запросы (через интернет)
                  ▼
┌─────────────────────────────────────────────────────┐
│  Сервер (удалённый компьютер)                       │
│                                                     │
│  ┌─────────────────────────────────────────┐        │
│  │ monitoring.jar (Java-приложение)        │        │
│  │                                         │  ← ЭТО │
│  │  Проверяет пароли                       │  БЭКЕНД│
│  │  Ходит в MongoDB / PostgreSQL           │        │
│  │  Возвращает данные (JSON)               │        │
│  │  + Отдаёт HTML/JS/CSS файлы             │        │
│  └─────────────────────────────────────────┘        │
└─────────────────────────────────────────────────────┘
```

**Фронтенд** — это HTML + CSS + JS файлы, которые **скачиваются** с сервера и **исполняются в браузере**. Сервер их только хранит и отдаёт, а вся "рисовальная" работа происходит на компьютере пользователя.

## В нашем НОВОМ проекте (wayrecall-tracker) мы это разделили

## Сценарий: Админ видит что-то, чего не должен

Допустим, у тебя есть права видеть только "Объекты" (vehicles), но не "Пользователи" (users). Что происходит?

### В старом legacy-проекте (монолит)

1. Ты заходишь, Spring Security проверяет твой JSESSIONID
2. При загрузке `index.html` **сервер уже знает кто ты и какие у тебя права**
3. Сервер может **сгенерировать JS-код** со ссылками только на доступные данные:
   ```javascript
   // В сгенерированном JS:
   var userHasObjectsRole = true;   // ← ты видишь
   var userHasUsersRole = false;    // ← ты НЕ видишь
   ```

**НО!** Злой админ может:
- Открыть DevTools (F12)
- Вручную вызвать `usersService.loadAllUsers()` 
- И попробовать загрузить список пользователей

### Вот тут включается вторая приходит БЭКЕНД:

В BillingAuthenticationProvider.scala и по всему коду везде наличествуют checks:

```scala
// Где-то в UserService.scala
```

**Процесс:**

3. JS в браузере сохраняет этот токен (обычно в localStorage)
4. Каждый XHR-запрос идёт с `Authorization: Bearer eyJhbGc...`
5. **API Gateway** проверяет подпись JWT (реальный ли токен?)
6. **device-manager** проверяет права и фильтрует данные

```json
{ "error": "Unauthorized" }
```

**С JWT но без прав UserView — бэкенд вернёт:**
```json
{ "error": "Forbidden: insufficient rights" }
```

## Итог

**Фронт НЕ отвечает за безопасность.** Он может показать UI, скрыть кнопки, не отправлять запросы. Но **любой юзер может открыть DevTools и попробать отправить запрос вручную.**

**Настоящая безопасность — на бэкенде.** Бэкенд никогда не доверяет UI, он всегда проверяет права перед тем как отдать данные.

Поэтому говорят: "Никогда не доверяй фронтенду" — там нельзя хранить секреты, пароли, ключи. Всё проверяется на сервере.

## Где это реализовано в нашем коде

Смотри API Gateway — это Scala приложение, которое:

2. **Получает каждый запрос**
3. **Проверяет JWT токен** (есть ли авторизация?)
4. **Маршрутизирует по правилам**:
   ```scala

   ```

5. **Добавляет метаданные** (user ID, org ID, permissions)
6. **Проксирует запрос** к нужному бэкенду
7. **Получает ответ** и возвращает браузеру

## Почему это хорошо?

| Вопрос | Ответ |
|---|---|
| **Безопасность** | API Gateway централизованно проверяет JWT, нельзя открыть порт device-manager напрямую |
| **Маршрутизация** | Если device-manager упадёт, API Gateway может вернуть ошибку или переслать на резервный |
| **Логирование** | Все запросы проходят через одну точку — легко логировать и анализировать |
| **Масштабирование** | Если device-manager перегружен, можно добавить ещё инстанц, API Gateway будет балансировать |
| **Изоляция** | Браузер не знает где живут бэкенды, все порты скрыты |

## JWT токен — что это вообще?

JWT (JSON Web Token) — это **закодированное письмо с подписью**. Представь:

```
"Привет, я Иван, мне 25 лет, я админ. — Подписано: Сервер"
```

Сервер кодирует это письмо так, чтобы **никто не мог подделать подпись**. Юзер носит это письмо в браузере и при каждом запросе показывает: "Смотрите, сервер сказал что я админ".

---

## Структура JWT: три части через точку

```
eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJzdWIiOiIxMjM0NTY3ODkwIiwibmFtZSI6IkpvaG4gRG9lIiwiaWF0IjoxNTE2MjM5MDIyfQ.SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c
```

Это три части (разделены точками):

### Часть 1: Header (заголовок)

```
eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9
```

Если декодировать (это base64, обычный текст но в шифре):

```json
{
  "alg": "HS256",     // алгоритм подписи (HMAC with SHA-256)
  "typ": "JWT"        // тип токена
}
```

### Часть 2: Payload (полезная нагрузка — данные)

```
eyJzdWIiOiIxMjM0NTY3ODkwIiwibmFtZSI6IkpvaG4gRG9lIiwiaWF0IjoxNTE2MjM5MDIyfQ
```

Декодировано:

```json
{
  "sub": "1234567890",           // subject (кто это?)
  "name": "John Doe",            // имя
  "iat": 1516239022,             // issued at (когда создан)
  "roles": ["ROLE_ADMIN"],       // роли
  "exp": 1516242622              // expiration (когда истекает)
}
```

### Часть 3: Signature (подпись)

```
SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c
```

Это **криптографическая подпись**, которая гарантирует что:
- Токен создал **именно сервер** (а не юзер)
- То, что внутри (Header + Payload) **не поменялось**

## Как создаётся JWT (на сервере)

```javascript
// Шаг 1: Юзер вводит логин/пароль
POST /login
{ username: "john@example.com", password: "1234" }

// Шаг 2: Сервер проверяет
if (passwordCorrect) {
  
  // Шаг 3: Сервер создаёт токен
  const header = { alg: "HS256", typ: "JWT" };
  const payload = {
    sub: "user123",
    name: "John Doe",
    roles: ["ROLE_USER"],
    iat: Date.now(),
    exp: Date.now() + 24*60*60*1000  // 24 часа
  };
  
  // Шаг 4: Сервер берёт СВОЙ секретный ключ
  const secret = "super-secret-key-that-only-server-knows";
  
  // Шаг 5: Подписывает
  const signature = HMAC_SHA256(
    header + "." + payload,
    secret
  );
  
  // Шаг 6: Склевает воедино
  const token = header + "." + payload + "." + signature;
  
  // Шаг 7: Отправляет браузеру
  return { token, expiresIn: 24*60*60 };
}
```

## Как сервер проверяет JWT

```javascript
// Сервер получил запрос с токеном
const token = request.headers['Authorization'].replace('Bearer ', '');
// token = 'eyJhbGc...eyJzdWI...SflKxw...'

// Шаг 1: Разбирает на три части
const [headerEnc, payloadEnc, signatureEnc] = token.split('.');

// Шаг 2: Проверяет подпись (КРИТИЧНЫЙ момент!)
const expectedSignature = HMAC_SHA256(
  headerEnc + "." + payloadEnc,
  secret  // ← ТОТ ЖЕ СЕКРЕТНЫЙ КЛЮЧ, что при создании!
);

if (signatureEnc !== expectedSignature) {
  // ✗ Подпись не совпадает — токен подделан!
  return 401 Unauthorized;
}

// Шаг 3: Проверяет время истечения
const payload = decode(payloadEnc);
if (payload.exp < Date.now()) {
  // ✗ Токен истёк
  return 401 Unauthorized;
}

// Шаг 4: Токен валиден!
currentUser = {
  id: payload.sub,
  name: payload.name,
  roles: payload.roles
};

// Обработка запроса с этим юзером
```

## Почему его нельзя подделать?

Допустим, юзер пытается:

```javascript
// Юзер в браузере пишет:
const fakePayload = {
  sub: "1234567890",
  name: "John Doe",
  roles: ["ROLE_ADMIN"]  // ← сам себе админ!
};

const fakeToken = header + "." + fakePayload + ".підписькакая-нибудь";
```

**Сервер проверяет:**

```javascript
const recalculatedSignature = HMAC_SHA256(
  header + "." + fakePayload,
  secret
);

// recalculatedSignature = "SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c" (правильная)
// но юзер отправил: "підписькакая-нибудь"
// Они не совпадают!

if (recalculatedSignature !== "підписькакая-нибудь") {
  return 401 Unauthorized;
}
```

**Почему не совпадают?** Потому что у юзера **нет секретного ключа**!

Подпись создаётся через **HMAC (Hash-based Message Authentication Code)** — специальный алгоритм, который:
- Берёт Header + Payload
- Смешивает с секретным ключом
- Выдаёт уникальный хеш

Без знания секретного ключа, даже если ты угадал подпись для одного токена, она не будет работать для другого, потому что Header/Payload разные.

Браузер должен:
1. Заметить что токен истёк
2. Отправить Refresh токен (если он есть)
3. Получить новый JWT
4. Повторить запрос

## Итог

**JWT токен — это:**
- ✓ Шифрованное письмо с подписью
- ✓ Браузер носит его с собой и показывает при каждом запросе
- ✓ Сервер проверяет подпись (убедиться что ЭТО сервер создал)
- ✓ Нельзя подделать без секретного ключа
- ✓ Имеет срок действия (expiresIn)

**Схема:**
```
Логин → сервер создаёт JWT → браузер сохраняет → 
браузер отправляет с каждым запросом → 
сервер проверяет → обрабатывает как авторизованный запрос
```

## 🛡️ На бэке должна быть защита (многоуровневая!)

### ✅ Уровень 1: JWT токен

API Gateway проверяет **подпись** токена:

```scala
// API Gateway (:8080)
def verifyToken(token: String): Task[JwtClaims] = for {
  // ПРОВЕРЯЕМ ПОДПИСЬ — подделать невозможно без SECRET KEY!
  claims <- ZIO.attempt(
    JWT.require(Algorithm.HMAC256(SECRET_KEY))  // ← SECRET хранится только на сервере
      .build()
      .verify(token)
  ).mapError(_ => InvalidToken)
  
  // Вытаскивают userId + organizationId из токена
  userId <- ZIO.attempt(claims.getSubject.toLong)
  orgId <- ZIO.attempt(claims.getClaim("orgId").asLong())
} yield JwtClaims(userId, orgId)
```

**Что пользователь не может сделать:** 
- ❌ Подделать токен (нет SECRET KEY)
- ❌ Изменить orgId внутри токена (сломается подпись)
- ❌ Подменить userId (сломается подпись)

### ✅ Уровень 2: API Gateway проверяет organization_id

```scala
// API Gateway
def listDevices(req: Request): Task[Response] = for {
  // 1. Вытаскиваем orgId из JWT
  orgId <- extractOrgIdFromToken(req)  // например, 123
  
  // 2. Получаем параметр из query
  queryOrgId <- req.queryParam("organization_id").map(_.toLong)  // пользователь вбил 999
  
  // 3. ПРОВЕРЯЕМ СОВПАДЕНИЕ
  _ <- ZIO.fail(if (orgId != queryOrgId) ForbiddenError("org mismatch"))
  
  // 4. Если совпадает, вызываем сервис с orgId из токена (НЕ из параметра!)
  result <- deviceService.listByOrganization(orgId)  // ← используем токен-orgId
} yield Response.json(result.toJson)
```

**Если пользователь попробует:**
```
GET /api/v1/devices?organization_id=999
Authorization: Bearer jwt_with_orgid_123
```
→ **API вернёт 403 Forbidden**, потому что `123 ≠ 999`

### ✅ Уровень 3: Database query обязательно фильтрует по organization_id

```scala
// Device Service
def listByOrganization(orgId: Long): Task[List[Device]] = for {
  // ВСЕГДА добавляем WHERE clause по organization_id
  devices <- repository.query(
    sql"SELECT * FROM devices WHERE organization_id = $orgId"
      .query[Device]
      .to[List]
  )
} yield devices
```

Даже если Gateway забыл проверить → БД всё равно вернёт только свои машины.

## 🚨 Где может быть дыра?

### ❌ Ошибка 1: Trust the frontend

```scala
// ❌ НЕПРАВИЛЬНО
def listDevices(req: Request): Task[Response] = for {
  orgId <- req.queryParam("organization_id").map(_.toLong)  // Берём прямо из параметра!
  result <- deviceService.listByOrganization(orgId)  // Даже не проверили токен!
} yield Response.json(result.toJson)
```

Пользователь вбивает любой orgId → **получает чужих машин**.

**Вывод:** ВСЕГДА вытаскивайте organization_id **из JWT токена**, не из параметров запроса!

### ❌ Ошибка 2: Забыли WHERE по organization_id в БД

```sql
-- ❌ НЕПРАВИЛЬНО
SELECT * FROM devices WHERE id = $deviceId;
-- Вернёт вообще любой девайс, даже из другой организации

-- ✅ ПРАВИЛЬНО
SELECT * FROM devices 
WHERE id = $deviceId AND organization_id = $orgId;
```

### ❌ Ошибка 3: Роли/permissions не проверяют

```scala
// ❌ НЕПРАВИЛЬНО
def updateDevice(id: Long, data: Device): Task[Device] = for {
  // Просто обновляем, не проверяя права юзера
  updated <- repository.update(id, data)
} yield updated

// ✅ ПРАВИЛЬНО
def updateDevice(id: Long, data: Device, userId: Long): Task[Device] = for {
  // Проверяем: юзер вообще может редактировать?
  _ <- checkPermission(userId, id, "devices:edit")
  // Проверяем: машина принадлежит его organization?
  device <- repository.findById(id).someOrFail(DeviceNotFound(id))
  _ <- ZIO.fail(if (device.organizationId != userOrgId) ForbiddenError())
  // ПОТОМ обновляем
  updated <- repository.update(id, data)
} yield updated
```

```scala
      orgId = authContext.organizationId  // ← из токена!
      
      // 2. Читаем данные из body
      updateData <- req.body.asJson[UpdateDeviceRequest]
      
      // 3. Проверяем: машина принадлежит orgId?
      device <- deviceRepository
        .findById(id)
        .someOrFail(DeviceNotFound(id))
      
      // 4. КРИТИЧНО: проверяем organisation_id
      _ <- ZIO.fail(ForbiddenError()).when(device.organizationId != orgId)
      
      // 5. Проверяем права (может ли юзер редактировать?)
      _ <- authService.checkPermission(userId, "devices:edit")
      
      // 6. Обновляем
      updated <- deviceService.update(id, updateData, orgId)  // ← используем orgId из токена
      
    } yield Response.json(updated.toJson)
  }
)
```

## 2. Параметры поездок — отличное наблюдение! 🎯

Ты **абсолютно прав** — но тут есть важный нюанс. Давай разберём:

### ❌ НЕТ, Connection Manager НЕ использует эти настройки

Connection Manager делает **низкоуровневую** фильтрацию (из ARCHITECTURE_BLOCK1.md):

```scala
// Connection Manager фильтры — ГЛОБАЛЬНЫЕ, одинаковые для всех:
case class FilterConfig(
  minDistanceMeters: Double = 5.0,    // GPS шум (< 5 метров)
  minTimeSecs: Int = 5,               // Дубликаты (< 5 сек)
  maxSpeed: Double = 300.0,           // GPS глюк (> 300 км/ч)
  invalidCoords: Boolean = true       // Координаты 0,0
)
```

Это **технические** фильтры — убираем мусор от GPS-приёмника. Одинаковые для всех устройств.

### ✅ Пользовательские настройки поездок используются в ДРУГОМ месте

```
Мин. скорость = 1 км/ч      → это НЕ фильтр GPS точек!
Мин. время стоянки = 300 сек → это НЕ фильтр GPS точек!
```

Это **параметры АЛГОРИТМА построения отчётов** — они используются при:

```
GPS точки в TimescaleDB
    ↓
Запрос отчёта "Поездки" пользователем
    ↓
Analytics Service читает точки за период
    ↓
Применяет настройки ЭТОГО объекта:
    - скорость < 1 км/ч → стоянка
    - стоянка > 300 сек → фиксируем
    - поездка > 180 сек И > 400м → фиксируем
    ↓
Формирует отчёт: поездка 1, стоянка 1, поездка 2...
```

### Connection Manager vs Analytics Service

| Параметр | Connection Manager | Analytics Service |
|----------|-------------------|-------------------|
| `minDistanceMeters: 5m` | ✅ Фильтр GPS шума | — |
| `maxSpeed: 300 км/ч` | ✅ Фильтр глюков | — |
| `repTripsMinMovementSpeed: 1 км/ч` | — | ✅ Определение "стоит/едет" |
| `repTripsMinParkingTime: 300 сек` | — | ✅ Определение "стоянка" |

## 4. Auth Service — отдельно или в API Gateway?

**Рекомендация: встроить в API Gateway.** Вот почему:

| Фактор | Отдельный Auth Service | Встроенный в API Gateway |
|--------|----------------------|--------------------------|
| Сложность | 2 сервиса, IPC между ними | 1 сервис |
| Latency | +3-5 мс на каждый запрос (проверка токена) | 0 мс (проверка локально) |
| Deploy | 2 контейнера, health checks | 1 контейнер |
| Масштабирование | Нужно масштабировать оба | Только Gateway |
| MVP скорость | Дольше | Быстрее ✅ |

Для MVP идеально: **Auth middleware внутри API Gateway**:
```scala
// API Gateway
object AuthMiddleware:
  def apply(routes: Routes): Routes =
    routes @@ Middleware.bearerAuth { token =>
      JWT.verify(token, SECRET)  // ← локально, без сетевого вызова
        .map(claims => AuthContext(claims.userId, claims.orgId, claims.role))
    }
```

**Login/Register** тоже в Gateway:
```
POST /api/v1/auth/login    → Gateway сам проверяет credentials в PostgreSQL
POST /api/v1/auth/register → Gateway сам создаёт юзера
POST /api/v1/auth/refresh  → Gateway сам обновляет токен
```

Отдельный Auth Service имеет смысл позже, когда добавляем OAuth (Google/Yandex), SSO, LDAP.
